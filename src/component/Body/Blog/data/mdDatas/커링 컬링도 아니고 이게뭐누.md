# 커링?? 컬링도 아니고 이게뭐누

커링은 **“인자를 한 번에 다 받지 않고, 하나씩(또는 몇 개씩) 잘라서 받는 함수로 바꾸는 기법”**이야.

조금만 더 폼 나게 말하면:

> 다인자 함수 f(a, b, c)를f(a)(b)(c)처럼 “인자를 쪼개서 순차적으로 받는 함수들의 체인”으로 변환하는 것.
> 

---

## 1. 예제로 보는 커링

### 일반 함수

```jsx
function add(a, b) {
  return a + b;
}

add(2, 3); // 5

```

### 커링된 함수

```jsx
function curriedAdd(a) {
  return function (b) {
    return a + b;
  };
}

const add2 = curriedAdd(2);
add2(3); // 5
add2(10); // 12

```

여기서:

- `curriedAdd(2)`를 호출하면,
    
    **“항상 2를 더하는 함수”**가 하나 만들어져서 `add2`에 들어감.
    
- 이게 가능한 이유가 바로 **클로저** 때문이야.
    
    `a` 값을 함수가 계속 기억하고 있기 때문에(`a = 2`), 나중에 `b`만 받아도 계산 가능.
    

---

## 2. 커링의 핵심 아이디어

커링을 하면 함수가 이런 느낌으로 바뀜:

- 원래: `f(a, b, c)`
- 커링 후: `f(a)(b)(c)`

그리고 중간 단계에서:

```jsx
const fWithA = f(a);      // a를 고정한 함수
const fWithAB = f(a)(b);  // a, b를 고정한 함수

```

이렇게 “**일부 인자를 미리 넣어두고, 나중에 나머지 인자를 받는 함수**”를 만들 수 있음 → 이걸 **부분 적용(Partial Application)**이라고도 부름.

---

## 3. 커링을 왜 쓰냐?

### 3-1. 재사용성 & 가독성 ↑

```jsx
function multiply(a, b) {
  return a * b;
}

function curryMultiply(a) {
  return function (b) {
    return a * b;
  };
}

const double = curryMultiply(2);
const triple = curryMultiply(3);

double(10); // 20
triple(10); // 30
```

- `double`, `triple`처럼 **의미 있는 이름의 함수**를 쉽게 만들 수 있음.
- “매번 `multiply(2, x)` 쓰는 것”보다 훨씬 읽기 좋음.

### 3-2. 공통 인자 고정 (설정값, 환경값 고정)

```jsx
function sendLog(level) {
  return function (message) {
    console.log(`[${level}] ${message}`);
  };
}

const info = sendLog("INFO");
const warn = sendLog("WARN");
const error = sendLog("ERROR");

info("시스템 시작");
warn("메모리 사용량 높음");
error("서비스 다운!");
```

- `level` 같은 공통 인자를 **앞에서 한 번만 넣고**,
    
    뒤로는 메시지만 넘겨도 됨.
    
- 실무에선:
    - API base URL,
    - 공통 옵션,
    - 공통 헤더,
    - 공통 prefix 등
        
        이런 걸 커링으로 고정해두고 씀.
        

### 3-3. 함수형 스타일과 잘 맞음

예를 들어 배열 필터링:

```jsx
const isGreaterThan = min => x => x > min;

const gt10 = isGreaterThan(10);
[5, 11, 20].filter(gt10); // [11, 20]
```

- `isGreaterThan(10)`에서 `10`을 기억한 함수가 만들어지고,
- 그걸 `filter`에 그대로 넘길 수 있음.
- 이런 패턴이 Ramda, lodash/fp 같은 함수형 라이브러리에서 엄청 많이 나옴.

---

## 4. 커링 vs 부분 적용(Partial Application)

둘이 비슷해서 많이 헷갈리는데, 엄밀히는 이렇게 볼 수 있어:

- **커링(currying)**
    - 다인자 함수 `f(a, b, c)`를
        
        “항상 **인자 하나씩만** 받는 함수 체인”으로 바꾸는 것
        
        → `f(a)(b)(c)`
        
- **부분 적용(partial application)**
    - `f(a, b, c)`에 인자 일부를 미리 넣어서
        
        “인자가 덜 필요한 새로운 함수”를 만드는 것
        
        → `g(b, c) = f(10, b, c)` 같은 느낌
        

실제 JS 실무에서는 이 둘을 엄청 엄격하게 구분하기보다는,

“**일부 인자를 미리 고정해 두고, 나머지를 나중에 받는 패턴 전체**”를

대충 “커링”이라고 부르는 경우도 많음.

---

## 5. 커링과 클로저의 관계

- 커링된 함수는 **무조건 클로저**를 사용한다.
- 이유:
    - 첫 번째 호출에서 받은 인자(`a`)를
    - 두 번째, 세 번째 호출 시점까지 **기억**해야 하기 때문.

```jsx
function makeAdder(a) {
  return function (b) {
    // 이 안에서 a를 사용하므로 클로저
    return a + b;
  };
}
```

- `a`는 이미 평가가 끝난 스코프의 변수인데도,
- 내부 함수가 `a`를 참조해서 계속 사용 → **클로저 메커니즘**.

즉,

> 클로저는 “상태 기억”이라는 능력을 주고,
커링은 그 “상태 기억”을 이용해서 인자를 나눠서 받는 패턴이라고 보면 됨.
> 

---

## 6. 한 줄 요약

- **커링이 뭐냐?**
    
    → 여러 인자를 받는 함수를, 인자를 하나씩(or 일부씩) 순차적으로 받는 함수들로 쪼개는 기법.
    
- **왜 쓰냐?**
    - 공통 인자(설정/환경)를 미리 고정하고,
    - 나중에 필요한 값만 넣어 쓰기 위해
    - 재사용성, 가독성, 함수형 스타일에 유리
- **어떻게 가능하냐?**
    
    → 내부 함수들이 앞에서 받은 인자를 **클로저로 기억하기 때문**.
    

# 그렇다면 어떻게 쓰는데?

## 1. 공통 설정 고정하는 API 클라이언트 (baseUrl, 헤더 등)

### 👀 패턴 아이디어

- 1단계: 프로젝트/도메인별 공통 설정을 **고정**
- 2단계: 실제 API path + 옵션만 넣어서 호출

### 🔧 코드

```tsx
// 공통 타입 정의
type HttpMethod = "GET" | "POST" | "PUT" | "PATCH" | "DELETE";

interface BaseConfig {
  baseUrl: string;
  defaultHeaders?: Record<string, string>;
}

interface RequestOptions {
  method?: HttpMethod;
  headers?: Record<string, string>;
  body?: unknown;
  query?: Record<string, string | number | boolean | undefined>;
}

const createApiClient =
  (baseConfig: BaseConfig) =>
  async <T = unknown>(path: string, options: RequestOptions = {}): Promise<T> => {
    const url = new URL(path, baseConfig.baseUrl);

    // query string 처리
    if (options.query) {
      Object.entries(options.query).forEach(([key, value]) => {
        if (value !== undefined) url.searchParams.set(key, String(value));
      });
    }

    const res = await fetch(url.toString(), {
      method: options.method ?? "GET",
      headers: {
        "Content-Type": "application/json",
        ...baseConfig.defaultHeaders,
        ...options.headers,
      },
      body: options.body ? JSON.stringify(options.body) : undefined,
    });

    if (!res.ok) {
      // 여기서 공통 에러 핸들링
      throw new Error(`API Error: ${res.status} ${res.statusText}`);
    }

    return (await res.json()) as T;
  };

```

### 🧠 사용 예시 (Sprint 도구 / VTDM 스타일)

```tsx
// HMG VTDM용 기본 클라이언트
export const hmgApi = createApiClient({
  baseUrl: "/api/v1",
  defaultHeaders: {
    "X-Client": "HMG-VTDM",
  },
});

// SUMS 전용
export const sumsApi = createApiClient({
  baseUrl: "https://sums.internal.hmg",
});

// 사용
const projects = await hmgApi<ProjectDto[]>("/projects", {
  query: { page: 1, size: 20 },
});

const result = await sumsApi<ReprogramResult>("/reprogram", {
  method: "POST",
  body: { vin, ecuId, fileId },
});

```

📌 **여기서 커링 포인트**

- `createApiClient(baseConfig)` 호출 시 **baseConfig를 클로저에 저장**
- 그 이후로는 `hmgApi`, `sumsApi`는 **설정이 묶인 함수**가 됨
    
    → 매번 `baseUrl`, `defaultHeaders` 안 넣어도 됨

## 2. React 이벤트 핸들러에서 인자 넘길 때

### 👀 패턴 아이디어

- 테이블/리스트에서 행 클릭 시 `rowId`, `type`, `mode` 같은 걸 함께 넘기고 싶을 때
- `onClick={() => handleClick(row.id, 'DETAIL', true)}` 이렇게 매번 쓰면 지저분함
    
    → 커링으로 **“부분 적용된 핸들러”**를 만들어 둠
    

### 🔧 코드

```tsx
// 커링된 핸들러 팩토리
const createRowClickHandler =
  (rowId: string, mode: "DETAIL" | "EDIT") =>
  (e: React.MouseEvent<HTMLTableRowElement>) => {
    e.preventDefault();
    // 여기서 rowId, mode 모두 안전하게 접근 가능 (클로저)
    console.log("clicked row:", rowId, "mode:", mode);
    // 예: 라우팅
    // navigate(`/ecus/${rowId}?mode=${mode.toLowerCase()}`);
  };

```

### 🧠 사용 예시

```tsx
{rows.map(row => (
  <trkey={row.id}
    onClick={createRowClickHandler(row.id, "DETAIL")}
    className="cursor-pointer hover:bg-slate-50"
  >
    {/* ... */}
  </tr>
))}

```

장점:

- JSX 내 인라인 로직이 줄어서 **가독성↑**
- 커링 덕분에 `row.id`, `mode`를 **이미 캡처한 함수**를 넘겨 주기만 하면 됨

## 3. Zustand + 커링: selector / action factory

네가 자주 쓰는 패턴이랑 딱 맞는 부분.

### 3-1. selector를 커링으로 빼기

```tsx
// store
interface User {
  id: number;
  name: string;
  team?: string;
  role: "USER" | "ADMIN" | "SUPER_ADMIN";
}

interface UserState {
  users: User[];
}

export const useUserStore = create<UserState>()(set => ({
  users: [],
}));

// 🔑 커링된 selector 팩토리
const selectUserById =
  (id: number) =>
  (state: UserState) =>
    state.users.find(u => u.id === id);

```

사용:

```tsx
const user = useUserStore(selectUserById(1001));

```

- `selectUserById(1001)`이 **부분 적용된 함수**가 되고,
- Zustand는 나중에 여기에 `state`를 넣어 호출함.
- `id`는 커링 + 클로저로 이미 기억되어 있음.

### 3-2. action도 커링으로 만들기

```tsx
interface PermissionState {
  permissions: Record<string, boolean>;
  togglePermission: (key: string) => void;
  // ...
}

export const usePermissionStore = create<PermissionState>()(set => ({
  permissions: {},
  togglePermission: key =>
    set(state => ({
      permissions: {
        ...state.permissions,
        [key]: !state.permissions[key],
      },
    })),
}));

// 🔑 커링된 핸들러 팩토리
export const createPermissionToggleHandler =
  (permissionKey: string) =>
  () => {
    const toggle = usePermissionStore.getState().togglePermission;
    toggle(permissionKey);
  };

```

사용:

```tsx
<Button onClick={createPermissionToggleHandler("ADMIN_PAGE_ACCESS")}>
  관리자 페이지 접근 토글
</Button>

```

여기서 포인트:

- `permissionKey`는 **커링+클로저로 고정**
- Zustand의 `getState()`를 사용해서 **stale closure 없이** 항상 최신 상태/액션을 가져옴
    
    → 이전에 우리가 얘기했던 “useCallback + stale closure” 이슈를 피하는 패턴이랑 딱 연결됨.

## 4. 필터 / 정렬 유틸: 조합 가능한 커링 함수

대시보드, VTDM 파일 리스트 필터링 같은 곳에서 꽤 유용함.

### 🔧 코드

```tsx
interface FileMeta {
  id: string;
  name: string;
  team?: string;
  owner?: string;
  tags: string[];
}

// 단일 조건 predicate들
const byTeam =
  (team: string) =>
  (file: FileMeta) =>
    file.team === team;

const byOwner =
  (owner: string) =>
  (file: FileMeta) =>
    file.owner === owner;

const hasTag =
  (tag: string) =>
  (file: FileMeta) =>
    file.tags.includes(tag);

// 배열에 적용하는 고차 함수
const filterWith =
  <T>(predicate: (item: T) => boolean) =>
  (list: T[]) =>
    list.filter(predicate);

```

### 🧠 사용 예시

```tsx
const files: FileMeta[] = /* ... */;

// 1) “VTDM 팀 + owner: 남훈 + tag: J1979”
const complexPredicate = (file: FileMeta) =>
  byTeam("VTDM")(file) &&
  byOwner("남훈")(file) &&
  hasTag("J1979")(file);

const filteredFiles = filterWith(complexPredicate)(files);

```

필터 UI 만들 때:

- `byTeam`, `byOwner`, `hasTag`는 **조건 하나를 캡처한 커링 함수**
- `filterWith`는 그걸 배열에 적용하는 **또 다른 커링 함수**
- 이거 조합하면 필터 로직을 form 상태랑 매핑하기 쉬워짐

## 5. 폼 검증: “필드 이름 고정 + 값만 넣는” 커링 패턴

```tsx
type Validator = (value: string) => string | null;

// 필수값 검증기 팩토리
const required =
  (fieldLabel: string): Validator =>
  value =>
    value.trim() === "" ? `${fieldLabel}을(를) 입력해주세요.` : null;

// 길이 제한 검증기 팩토리
const maxLength =
  (fieldLabel: string, max: number): Validator =>
  value =>
    value.length > max
      ? `${fieldLabel}은(는) 최대 ${max}자까지 가능합니다.`
      : null;

// 여러 검증기 합성
const combineValidators =
  (...validators: Validator[]): Validator =>
  value => {
    for (const v of validators) {
      const err = v(value);
      if (err) return err;
    }
    return null;
  };

```

사용:

```tsx
const validateProjectName = combineValidators(
  required("프로젝트 이름"),
  maxLength("프로젝트 이름", 50)
);

const error = validateProjectName(form.name);
if (error) {
  toast.error(error);
}

```

- `required("프로젝트 이름")` 호출 시 라벨이 **커링+클로저로 고정**
- 나중에는 value만 넣어 호출
    
    → 폼 필드 많을 때도 검증 로직이 깔끔해짐

## 마무리 요약

방금 본 예제들 공통 패턴은 전부 이거야:

1. **첫 번째 호출에서 “환경/설정/키/ID”를 고정**
2. 그 상태를 **클로저로 기억**하는 함수(커링된 함수)를 반환
3. React/Zustand/유틸에서 필요할 때마다 그 함수를 그냥 전달하거나 호출
