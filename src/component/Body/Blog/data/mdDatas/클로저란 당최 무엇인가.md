# 클로저란 당최 무엇인가

## 1. 클로저란 정확히 무엇인가?

보통 이렇게 말하죠.

> “클로저는 반환된 내부 함수를 의미한다.”
> 
> 
> “내부 함수가 자신이 선언됐을 때의 어휘적(렉시컬) 환경을 기억해서, 바깥 스코프 밖에서 호출돼도 그 스코프에 접근할 수 있는 것.”
> 

이 말은 거의 맞는데, 전문가 관점에서 조금 더 정확히 정리하면:

> 클로저(closure)는 “함수(Function)” + “그 함수가 선언될 당시의 렉시컬 환경(Lexical Environment)”을 묶어놓은 런타임 객체입니다.
> 
- “반환된 내부 함수”는 **클로저가 만들어지는 전형적인 패턴**일 뿐이고,
- 꼭 `return`으로 내보내지 않아도, **외부 스코프 변수를 참조하는 모든 함수**는 클로저가 됩니다.
    
    (예: 이벤트 핸들러, 콜백, setTimeout 안의 함수 등)
    

즉, **함수가 살아 있는 한, 그 함수가 참조하던 스코프도 “필요한 부분만” 같이 살아남는 구조**가 바로 클로저예요.

---

## 2. 렉시컬 스코프 & 어휘적 환경(Lexical Environment)

### 2-1. 렉시컬 스코프(lexical scope)

- **“함수가 어디서 선언되었느냐”로 스코프가 결정되는 것**이 렉시컬 스코프입니다.
- 런타임에 “어디서 호출되었는지”가 아니라, **코드 작성 시점의 위치**가 기준입니다.

```jsx
const x = 10;

function outer() {
  const y = 20;

  function inner() {
    console.log(x, y); // 10, 20
  }

  return inner;
}

```

`inner` 함수는 **코드 상에서** `outer` 안에 선언되어 있으므로,

`outer`의 지역 변수 `y`와 전역 변수 `x`에 접근할 수 있습니다.

이 “선언 위치 기반의 규칙”이 렉시컬 스코프.

### 2-2. 어휘적 환경(Lexical Environment) 구조

JS 엔진 관점에서 보면 렉시컬 환경은 대략 이렇게 구성됩니다:

1. **Environment Record**
    - 현재 스코프에서 선언된 식별자(변수/함수/매개변수) → 실제 값 매핑
    - 예: `{ x: 10, y: 20 }`
2. **Outer Environment Reference**
    - “바깥 스코프의 렉시컬 환경”에 대한 포인터
    - 이 포인터들을 따라가는 **체인**이 우리가 알고 있는 *스코프 체인(scope chain)*

즉, 내부 함수에서 어떤 식별자를 찾을 때:

1. 자기 Environment Record에서 찾고,
2. 없으면 Outer Environment로 올라가고,
3. 거기에도 없으면 그 바깥으로…
    
    이렇게 올라가다가 끝까지 못 찾으면 `ReferenceError`.
    

이 “Environment Record + Outer 참조” 묶음이 바로 **렉시컬 환경 객체**이고,

클로저는 **이 객체에 대한 참조를 내부 함수와 함께 유지**하는 구조입니다.

---

## 3. 클로저의 동작 원리 (메모리 관점까지)

일반적인 지역 변수는 함수 호출이 끝나면 스택에서 사라진다고 배웠을 거예요.

그런데 클로저가 참조하는 변수는 어떻게 살아남을까요?

1. `outer()`가 호출되면:
    - `y` 같은 지역 변수는 원래라면 호출 종료 후 사라져야 할 “지역 스코프”에 있습니다.
2. 그런데 `inner` 함수가 `y`를 참조하고 있고,
    
    이 `inner` 함수가 바깥으로 반환(return)되어, 다른 곳에서 계속 사용됩니다.
    
3. JS 엔진의 GC는 “**아직 참조되고 있는 값은**” 메모리에서 제거하지 않습니다.
    - `inner`가 `y`가 들어있는 렉시컬 환경을 **참조**하고 있기 때문에,
    - 그 환경 객체는 **힙(Heap)** 에 살아남고, 필요할 때마다 다시 접근됩니다.
4. 나중에 `inner`에 대한 모든 참조가 끊기면 (예: 변수에 `null` 대입, 스코프에서 벗어남 등),
    - 그제야 GC가 “이 환경은 더 이상 누구도 참조 안 하네?” 하고 제거합니다.

즉:

> 함수 호출이 끝나도, 그 함수의 지역 변수를 참조하는 다른 함수(클로저)가 남아 있으면,
> 
> 
> 그 지역 변수들이 포함된 렉시컬 환경은 힙에 유지됩니다.
> 

이게 바로 “외부 함수가 리턴된 이후에도, 내부 함수가 외부 함수의 변수에 접근할 수 있다”라는 말의 **정확한 이유**입니다.

## 4. 클로저를 왜 사용하는가? (실전 용도)

### 4-1. 정보 은닉 / 캡슐화 (가장 대표적인 목적)

```jsx
function createCounter() {
  let count = 0; // 외부에서 직접 접근 불가 (private처럼 동작)

  return {
    inc() {
      count++;
      return count;
    },
    get() {
      return count;
    }
  };
}

const counter = createCounter();
counter.inc(); // 1
counter.inc(); // 2
counter.get(); // 2
// 외부에서 count 직접 수정 불가능

```

- `count`는 **오직 `inc`, `get` 클로저를 통해서만 접근 가능**합니다.
- 이런 패턴은 클래스를 사용하지 않고도 **“private 상태 + 공개 메서드”**를 구현할 수 있게 합니다.

### 4-2. 상태가 있는 함수 / 커링(curried) 함수

```jsx
function makeAdder(x) {
  return function (y) {
    return x + y;
  };
}

const add10 = makeAdder(10);
add10(5);  // 15
add10(20); // 30

```

- `add10`은 “항상 10을 더하는 함수”라는 **상태를 가진 함수**입니다.
- 이런 패턴은:
    - 커링(curring)
    - 포인트프리 스타일
    - 고차 함수(Higher-Order Function)
        
        같이 함수형 패턴에서 자주 쓰입니다.
        

### 4-3. 비동기/콜백 환경에서 변수 고정하기

```jsx
for (var i = 0; i < 3; i++) {
  (function (iCopy) {
    setTimeout(() => console.log(iCopy), 100);
  })(i);
}
// 0 1 2

```

- 즉시 실행 함수(IIFE)가 `iCopy`를 캡처해서,
    
    각 타이머마다 **당시의 i 값**을 “고정”합니다. 이게 전형적인 클로저 응용.
    

(ES6 이후에는 `let`으로 블록 스코프를 쓸 수 있어서 이런 패턴은 줄었지만, **원리는 여전히 클로저**입니다.)

### 4-4. React / Hooks, Zustand 등에서의 활용

- `useState`, `useEffect` 같은 Hook 역시 내부적으로 **클로저를 강하게 활용**합니다.
    - 컴포넌트가 다시 렌더링될 때도 이전 상태에 접근하는 구조 자체가 클로저 기반.
- Zustand도 `create(store => { ... })` 내부에서 정의한 함수들이
    
    store 상태를 참조하는 방식으로 동작 → 이것도 전형적인 클로저 패턴.

## 5. 자주 헷갈리는 포인트 정리

### 5-1. “반드시 return 해야만 클로저인가요?”

아니요.

```jsx
let saved;

function outer() {
  let x = 10;
  function inner() {
    console.log(x);
  }
  saved = inner; // 외부 변수에 저장만 해도 클로저가 됨
}

outer();
saved(); // 10

```

- “**외부 스코프의 변수를 참조하는 함수가, 외부 스코프보다 오래 살아있는 상황**”이면 모두 클로저입니다.
- 반환(return)은 그 중 가장 흔한 방법일 뿐이에요.

### 5-2. 클로저 = 무조건 보안/캡슐화용?

- **주 용도 중 하나**가 “캡슐화/정보은닉”인 건 맞습니다.
- 하지만 클로저는 더 넓게 보면:
    - **상태를 가진 함수**
    - **부분 적용 / 커링**
    - **콜백에서의 값 캡처**
    - **라이브러리 내부 구현 디테일 숨기기**
        
        같은 다양한 패턴의 기반이 되는 “언어 레벨 메커니즘”입니다.
        

### 5-3. 클로저의 부작용 / 주의점 (전문가 관점에서 중요한 부분)

1. **불필요한 메모리 유지**
    - 클로저가 많은 변수들을 참조하면,
        
        그 변수들이 GC 대상에서 제외되어 **생각보다 오래 메모리를 차지**할 수 있습니다.
        
    - 특히 대형 객체를 잡고 있을 때 주의.
2. **stale closure 문제**
    - React, Zustand에서 자주 겪는 이슈:
        - `useCallback`에 상태를 의존성에 안 넣으면,
            
            **예전 상태를 계속 캡처한 함수**가 남아서 “상태가 안 바뀐 것처럼 보이는” 현상 → 전형적인 클로저 문제.
            
    - 해결책이 “의존성 배열에 넣어라 / 클로저를 끊고 getState로 직접 가져와라”였던 거, 이미 경험하셨을 듯.
3. **디버깅 난이도 증가**
    - 어디서 어떤 클로저가 어떤 값을 캡처하고 있는지,
        
        코드가 복잡해질수록 추적이 어려워질 수 있습니다.
        
    - 그래서 팀 코드에서는 **클로저를 과하게 중첩시키기보다**, 적당히 분리된 모듈/클래스/훅으로 정리하는 게 중요.

## 6. 한 번에 요약

- **클로저란?**
    
    → **함수 + 그 함수가 선언된 시점의 렉시컬 환경을 함께 기억하는 런타임 객체.**
    
- **어떻게 동작하나?**
    
    → 함수가 외부 스코프의 변수를 참조하면, JS 엔진은 그 스코프의 렉시컬 환경을 힙에 유지하고,
    
    그 함수(클로저)가 필요할 때마다 그 환경을 따라 올라가서 값에 접근한다.
    
- **왜 쓰나?**
    - 외부에서 마음대로 건드릴 수 없는 “private 상태” 만들기 (정보 은닉, 캡슐화)
    - 상태를 가지는 함수, 부분 적용, 커링, 콜백에서 값 고정
    - React Hooks, Zustand, 라이브러리 내부 구현 등 현대 JS 프레임워크의 핵심 메커니즘
- **주의할 점**
    - 불필요한 변수 캡처로 인한 메모리 낭비
    - stale closure로 인한 “상태 안 바뀌는 것처럼 보이는” 버그
    - 디버깅 난이도
